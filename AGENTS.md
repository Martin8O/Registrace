# Registrace — project rules

Bilingual (CZ/EN) event-registration app for Diamond Way Buddhism (BDC) centres.
Next.js 16 App Router · TypeScript strict · Prisma 7 · Supabase (Postgres + Auth) ·
next-intl 4 · Zod 4 · Resend · Vercel. **No Docker.**

Read [`README.md`](README.md) for the architecture and [`CLAUDE.md`](CLAUDE.md) for the full
list of invariants before changing anything structural.

## Commands

```bash
npm run dev                 # dev server on :3000
npm run build               # production build
npm test                    # Vitest (81 tests, no database needed)
npm run lint                # ESLint
npx prisma migrate deploy   # apply migrations (needs DIRECT_URL)
```

## Non-negotiable rules

These are enforced across the codebase — do not violate them to make something work:

1. **Auth = Supabase Auth. Data = Prisma. Never mix.** The Supabase JS client is for Auth
   only; it must never read or write tables.
2. **The pricing engine (`modules/pricing`) is pure, server-only, and has no DB access.**
3. **Frontend prices are informational.** The server always recomputes the authoritative
   price before any DB write.
4. **No fat route handlers** — business logic lives in `modules/*`. Server Components call
   those services directly rather than fetching the app's own API.
5. **UI text → next-intl JSON** (`locales/*.json`); **event content → bilingual DB columns**
   (`*_cs` / `*_en`).
6. **Email failure never rolls back** the registration transaction.
7. **Soft delete (`deletedAt`)** on audit-relevant entities — no permanent deletion.
8. **Money = whole-CZK integers** (80 CZK is `80`). **Datetimes = UTC in the DB**, rendered in
   Europe/Prague.
9. **SUPER_ADMIN sees all; ADMIN is scoped to their centre(s).** An *owner* tier guards
   super-admin management.
10. Fields named `*Discount` are **subtracted** from the total, not added.

## Things that will surprise you

- **There is no `middleware.ts`.** Edge logic lives in `proxy.ts`. Its `config.matcher` covers
  pages and `/api/admin/**` **only** — the public API (`/api/events`, `/api/registration/**`,
  `/api/auth/me`) bypasses the edge entirely and rate-limits itself inside each handler.
- **The Prisma client is generated to `generated/prisma`** (gitignored, regenerated by the
  `postinstall` hook) — not to `node_modules/.prisma`.
- **Supabase needs two connection strings**: pooled `DATABASE_URL` (6543) at runtime, direct
  `DIRECT_URL` (5432) for migrate/seed. Migrations cannot run over the pooler.
- **Pricing is data-driven.** No age is hard-coded to 0 — every age is charged by its
  `PricingRule.dailyRate`. Arrival/departure discounts apply to 15+ only because child rules
  carry `0` discounts, not via an age branch.
- **RLS is enabled deny-all** on the data tables as a backstop, but Prisma connects directly
  and bypasses it. The real authorization is the role/ownership gate in the handlers and
  services.
- **Admin passwords never touch our server.** set-password and profile call Supabase Auth
  straight from the browser, so `lib/validation/password` is informational (same status as
  frontend pricing) and the enforcing policy lives in the Supabase dashboard. Changing one
  without the other silently desyncs them.

## Conventions

- Translation keys are **nested**, never flat. Namespaces: `form`, `home`, `event`, `badge`,
  `admin`. Keep `form.pricing_info` (in-form label) and `event.pricingInfo` (detail-page
  button) distinct — different elements, never merge.
- Validation lives in `lib/validation/*` and must stay **client-safe** (no Prisma imports).
- Change the schema only via a Prisma migration (driver-adapter pattern; `DIRECT_URL`).
